## Recruiter takeaway

YADL is the clearest proof that I can turn design-system judgment into productized tooling.

The project is not just "I made a Figma plugin." It shows that I can spot a repeated design operations problem, define the quality rules, design the plugin UX, build the Figma API implementation, and ship a public tool that other teams can use without my involvement.

For a product design, design systems, or UX engineering role, the important signal is this:

- I think about design quality as a system, not a personal review habit
- I can translate subjective UI feedback into concrete checks
- I understand the messy handoff point between Figma, tokens, components, and implementation
- I can design internal tools that reduce repeated manual work
- I can move from product framing to interface design to front-end implementation

## Problem

Design systems are only useful when designers actually apply them consistently.

In practice, that breaks down quickly. Files get inherited, copied, edited under deadline pressure, and filled with slightly wrong spacing, hard-coded colors, detached styles, generic layer names, hidden layers, and broken variables. Each issue is small on its own. Together, they create noisy handoffs and make UI quality depend on whoever has time to review the file manually.

The problem I wanted to solve was simple:

How do you make design-system QA repeatable inside Figma, before the file reaches development?

## System model

YADL works like a design linter for selected Figma layers.

The designer selects layers, runs the plugin, reviews grouped findings, and applies or dismisses suggestions. Instead of asking someone to remember every naming, token, spacing, and component rule by hand, the plugin turns those checks into a visible workflow.

The current check areas cover:

- Broken variables

## Plugin experience

The UX is deliberately direct: select layers, analyze, review suggestions.

The interface groups issues by category so the designer can focus on one type of cleanup at a time. In the captured demo, the error view shows 33 total findings, grouped into categories like spacing, color, radius, and stroke. YADL also separates immediately available fixes from findings that need review, so the plugin supports both quick cleanup and more deliberate design decisions.

The variables view exposes collections, modes, and variable counts. That matters because design-system issues are not only visual. They often come from token and library drift: the design looks plausible, but the underlying variable reference is wrong, missing, or inconsistent with the intended system.

## Rule engine

The technical challenge was making Figma document analysis feel lightweight.

Figma plugins run with a split architecture: the UI runs separately from the plugin sandbox that can access the Figma document. YADL uses that separation intentionally:

- The sandbox traverses the selected node tree and evaluates each node against the enabled checks
- The UI presents grouped findings, filters, smart suggestions, and apply actions
- Rule modules stay independent so checks can be expanded without turning the plugin into one tangled audit function
- Hidden layers and irrelevant branches can be skipped to avoid wasting time on noise
- Properties such as fills, strokes, typography, effects, layout spacing, and variables are normalized into comparable rule inputs

That architecture is the interesting part. It turns subjective review patterns into a small, extensible quality system.

## Product decisions

I made YADL free and public because design-system quality problems are not limited to enterprise teams.

Figma has since introduced its own "Check designs" feature for Organization and Enterprise plans, which validates the problem space: design QA is becoming a platform-level workflow, not just a personal best practice. YADL sits in that same problem category, but as a lightweight public plugin that teams can try without enterprise access.

The design goal was not to replace designer judgment. It was to remove the boring checks so judgment can be spent on the parts that actually need a human.

## Impact

YADL is now a public Figma Community plugin used by 22+ teams.

The strongest impact is not just the install count. It is the kind of work the plugin replaces: repeated manual review, subjective cleanup comments, and avoidable design-system drift before handoff.

For my portfolio, the project proves a useful combination:

- Product thinking: identifying a repeated workflow problem with clear user value
- Design systems: understanding tokens, variables, components, and consistency at scale
- UX design: making the audit flow scannable and action-oriented inside a constrained plugin surface
- Engineering: building against the Figma plugin API with a modular rule engine
- Design operations: turning standards into a repeatable workflow other people can run

That is why YADL belongs next to the larger product case studies. It is small in surface area, but strong in signal.
