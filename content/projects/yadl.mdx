## Why this matters

YADL is a good example of the work I like doing: take a messy design-system problem, make it understandable, and ship the tool that helps people fix it.

The plugin started with a practical handoff problem. Figma files often drift away from the system they are supposed to follow, especially when teams copy old frames, move fast, or inherit messy files. I turned that into a public Figma plugin used by 400+ teams globally.

![YADL product tagline: Less pixel policing. More consistency.](/projects/yadl/hero-tagline.jpg)

The project crosses the parts of design engineering I want to be judged on:

- Product judgment around a real design workflow
- Design-system knowledge across Figma variables, tokens, components, and handoff
- Implementation depth in the Figma plugin API, React, async document traversal, and product polish

## Outcome

YADL is a shipped design-system QA tool for Figma. It helps teams find token drift, hard-coded values, missing component documentation, broken variables, hidden or locked layer issues, and other cleanup work that often shows up as handoff feedback.

The useful part is the workflow it replaces: repeated pixel-policing, subjective cleanup comments, and preventable design-system drift before handoff.

## What YADL does

YADL focuses on the problems design-system teams actually review:

- Token and variable enforcement for padding, gap, radius, stroke width, fills, strokes, effects, typography, and broken variable references
- Smart fixes and bulk cleanup for matching variables, large selections, and undoable applied suggestions
- Component and library hygiene for generic layer names, nested components, absolute positioning, slot internals, and missing component documentation
- Workflow polish through selection-aware checks, configurable hidden and locked layer behavior, strict mode, keyboard shortcuts, persistent settings, and variable collection control

## Recent product iteration

The latest release work made YADL feel more like a product and less like a debug panel.

Recent versions added the new demo video and product images, refreshed the interface with animated icons and the Open Runde typeface, made re-linting stable after repeated scans, added component documentation checks, improved variable and style previews, and fixed resizing and narrow-window layout issues.

That matters because this plugin lives inside Figma's constrained plugin panel. The rule engine can be solid, but the tool still fails if designers cannot read it, resize it, or trust it during cleanup.

## Problem

Design systems only work when their rules survive normal production pressure. Inherited files, copied frames, deadlines, hidden layers, locked layers, detached styles, and one-off values all create drift.

Manual QA does not scale because every review depends on someone noticing the same tiny problems again.

The product question was: how can design-system QA happen inside Figma, before the file reaches development, without asking designers to become pixel police?

## My role

Solo Design Engineer. I owned product framing, plugin UX, Figma plugin implementation, rule design, copy, visual polish, and public release.

I treated YADL as a product rather than an automation script. That meant defining the workflow, making the findings understandable, creating useful fix paths, and keeping the UI calm enough to use during real design cleanup.

## Product decisions

- Keep linting selection-based so designers can audit a component, frame, or flow without scanning an entire file.
- Sort fixable errors above manual ones so the next action is obvious.
- Present smart suggestions as reviewable and reversible actions, not silent automation.
- Let teams configure hidden layers, locked layers, strict mode, component slots, and variable collections.
- Show variable collections, modes, style counts, and token previews because design-system drift is often structural as well as visual.

## Plugin experience

The UX is deliberately direct: select layers, analyze, review suggestions.

The error list groups issues by type and value, separates manual fixes from smart suggestions, and makes scope clear when hidden or locked layers are involved. Color suggestions respect opacity, so 10% black is not treated like solid black. Labels and cards scale across small and large plugin widths so the tool stays usable on laptop screens.

The variables view exposes collections, modes, saved libraries, enabled collections, style counts, and token previews. This gives designers a practical way to understand which tokens are available before applying fixes.

## Technical proof

Figma plugins run with a split architecture: the UI runs separately from the plugin sandbox that can access the Figma document. YADL uses that separation intentionally.

- The sandbox traverses the selected node tree and evaluates each node against enabled checks
- The React UI presents grouped findings, filters, variable data, smart suggestions, and apply actions
- Backend message handlers cover linting, selection, settings, variables, styles, saved libraries, cache management, and suggestion application
- Rule modules stay independent across structure, layout, visual, typography, and variable linting
- A variable/style index maps collections, modes, scopes, saved library data, and deleted references into fast suggestion lookups
- Async processing and progress sessions keep longer audits responsive
- Relint stability fixes prevent repeated scans from freezing the plugin

Together, those pieces let the plugin run the same review patterns repeatedly instead of relying on someone to remember every rule by hand.
