## Recruiter takeaway

YADL is a productized design-system QA workflow, not just a plugin experiment.

The public Figma plugin is used by 400+ teams globally. It automates the boring part of design-system enforcement: finding rogue spacing, hard-coded colors, detached styles, missing component documentation, broken variables, and other quality issues before they reach development.

The core loop is intentionally simple: select layers, scan the selection, review grouped findings, inspect smart suggestions, and clean up issues before developer handoff.

![YADL product tagline: Less pixel policing. More consistency.](/projects/yadl/hero-tagline.jpg)

What this proves:

- I can turn subjective design-system quality into concrete, repeatable checks and fixes
- I understand the handoff point between Figma, tokens, components, and implementation
- I can frame, design, build, polish, and ship a public design tool end-to-end

## Problem

Design systems are only useful when designers actually apply them consistently. Otherwise they become documentation instead of infrastructure.

In practice, that breaks down quickly. Files get inherited, copied, edited under deadline pressure, and filled with slightly wrong spacing, hard-coded colors, detached styles, generic layer names, hidden layers, locked layers, and broken variables. Each issue is small on its own. Together, they create noisy handoffs and make UI quality depend on whoever has time to review the file manually.

The problem I wanted to solve was simple:

How do you make design-system QA repeatable inside Figma, before the file reaches development, without asking designers to become pixel police?

## My role

Solo **Design Engineer** — owned product framing, plugin UX, Figma plugin implementation, rule design, copy, visual polish, and public release.

## Constraints

- Figma plugin UI and document access run in separate contexts
- Findings had to be understandable without turning the plugin into a noisy checklist
- Rules needed to support real design-system cleanup without pretending every team has identical standards
- The interface had to stay usable inside Figma's small, resizable plugin window

## Key decisions

- Group findings by issue type and value so designers can scan patterns instead of reading a raw error stream
- Use smart suggestions where the plugin can identify likely fixes, while leaving final judgment visible and reversible
- Keep the workflow selection-based so designers can audit a frame, flow, or component without scanning an entire file
- Make variable collections and modes visible, because token drift is often the real source of visual inconsistency
- Treat responsive plugin layout as a product requirement, not polish, so the tool works in narrow and wide Figma panels

## Outcome

YADL is now a public Figma Community plugin used by 400+ teams globally, with 60+ frequent daily users. It has a small surface area, but strong signal: design systems, tooling, implementation, product judgment, and handoff quality in one shipped artifact.

## What it checks

YADL works like a design-system enforcer for Figma files. It catches the kinds of small inconsistencies that usually become manual review comments:

- Variable-aware linting for padding, gap, radius, stroke width, fills, strokes, effects, and typography
- Auto-suggestions and bulk apply for matching design tokens across large selections
- Deleted variable detection, with one-click migration when an exact replacement exists
- Generic layer names, absolute positioning, nested components, and slot internals
- Hidden and locked layers, configurable per team workflow
- Component and variant documentation checks for missing descriptions or documentation links
- Keyboard-first flows: `Enter` to analyze, `Cmd+R` to relint, `?` for shortcuts, and full keyboard navigation
- Strict mode, variable collection control, undo, persistent settings, and selection-aware Apply All

## System model

YADL works like a design linter for selected Figma layers.

The designer selects layers, runs the plugin, reviews grouped findings, and applies or dismisses suggestions. Instead of asking someone to remember every naming, token, spacing, style, and component rule by hand, the plugin turns those checks into a visible workflow.

The current check areas cover:

- Generic layer names and missing component descriptions
- Typography and text-style consistency
- Padding and gap
- Stroke color and stroke width
- Fill colors
- Effects and shadows
- Border radius, including individual corners
- Locked layers
- Hidden layers
- Nested components
- Slot/component internals
- Broken variables

## Plugin experience

The UX is deliberately direct: select layers, analyze, review suggestions.

The latest interface focuses on scanability. Fixable errors rise to the top, manual issues are separated, and grouped cards show the value that needs attention. Color suggestions now respect opacity, so 10% black no longer gets treated like solid black.

Recent UI work also made the plugin more resilient in real Figma use: labels scale across default, small, and large sizes; the resize handle no longer collapses the layout; and the same linting workflow remains readable in both narrow and wide plugin windows.

The variables view exposes collections, modes, saved libraries, style counts, enabled collections, and token previews. That matters because design-system issues are not only visual. They often come from token and library drift: the design looks plausible, but the underlying variable reference is wrong, missing, or inconsistent with the intended system.

## Rule engine

The technical challenge was making Figma document analysis feel lightweight.

Figma plugins run with a split architecture: the UI runs separately from the plugin sandbox that can access the Figma document. YADL uses that separation intentionally:

- The sandbox traverses the selected node tree and evaluates each node against the enabled checks
- The React UI presents grouped findings, filters, smart suggestions, variable data, and apply actions
- The backend routes plugin messages through dedicated handlers for linting, selection, settings, variables, styles, saved libraries, cache management, and suggestion application
- Rule modules stay independent across structure, layout, visual, typography, and variable linting, so checks can be expanded without turning the plugin into one tangled audit function
- Hidden layers and irrelevant branches can be skipped to avoid wasting time on noise
- Properties such as fills, strokes, typography, effects, layout spacing, styles, and variables are normalized into comparable rule inputs

That architecture turns subjective review patterns into a small, extensible quality system.

The source also shows the product-level engineering work behind the visible UI:

- A variable/style index maps Figma collections, modes, scopes, saved library data, and deleted references into fast suggestion lookups
- Smart suggestion handlers match issue types to likely fixes for spacing, radius, stroke width, typography, effects, deleted variables, and default names
- Async processing and progress sessions keep longer audits responsive instead of freezing the plugin surface
- Relint stability fixes prevent repeated scans from freezing the plugin
- Bulk apply, undo, dismiss, retry, and selection-restore flows make the plugin useful for real cleanup work, not just reporting

## Product decisions

I made YADL free and public because design-system quality problems are not limited to enterprise teams.

Figma has since introduced its own "Check designs" feature for Organization and Enterprise plans, which validates the problem space: design QA is becoming a platform-level workflow, not just a personal best practice. YADL sits in that same problem category, but as a lightweight public plugin that teams can try without enterprise access.

The design goal was not to replace designer judgment. It was to remove the boring checks so judgment can be spent on the parts that actually need a human.

That shaped several implementation choices:

- Settings let teams decide which checks matter for their workflow instead of forcing one rigid definition of quality
- Strict mode supports production-ready audits without making every casual file feel noisy
- Hidden, locked, and nested component checks are configurable because those layers can be signal in one team and noise in another
- Smart suggestions are presented as reviewable actions, not silent automation
- The icon system, type treatment, and variable preview redesign make the plugin feel like a product rather than a debug panel

## Impact

The strongest impact is not just the install count. It is the kind of work the plugin replaces: repeated manual review, subjective cleanup comments, and avoidable design-system drift before handoff.

- UX design: making the audit flow scannable and action-oriented inside a constrained plugin surface

That is why YADL belongs next to the larger product case studies.
