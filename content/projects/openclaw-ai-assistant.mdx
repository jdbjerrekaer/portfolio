## Recruiter takeaway

This project is not here to say "I built a chatbot."

It shows that I can turn AI into a reliable product workflow: define the system boundaries, design the interaction model, build the supporting scripts and content structure, test failure cases, and keep risky actions under human control.

For a Danish product design, UX engineering, or design systems role, that is the relevant signal:

- I design AI-enabled workflows around real user constraints, not novelty
- I structure ambiguous operational work into reusable systems
- I can move between UX framing, implementation, documentation, and release verification
- I think in guardrails, handoffs, states, and feedback loops
- I use automation to reduce repeated work while keeping accountability visible

The strongest part of the project is the operating model. OpenClaw is useful because the assistant knows what it may do, what it must remember, what it must never expose, and when a human decision is required.

## Problem

Most AI tools are useful for a single interaction, then forget the work as soon as the chat ends.

That is fine for quick questions. It is weak for recurring work: maintaining a self-hosted setup, preparing social media material from messy source files, checking scheduled follow-ups, remembering project constraints, and turning repeated fixes into repeatable workflows.

The design challenge was not "how do I use more AI?"

It was: how do I make AI useful enough for repeated work while keeping trust, review, and recovery visible?

## Operating model

OpenClaw is the runtime. JARVIS is the assistant layer I use inside it.

Together, they form a self-hosted workspace for coordinating messages, files, scripts, memory, skills, scheduled jobs, and project workflows. Most interactions happen through Telegram, where the assistant can receive requests, send status updates, ask for approval, or deliver finished work.

The system runs on a simple loop:

1. A request arrives through chat, a schedule, or a project trigger.
2. The assistant reads the relevant memory, docs, files, and skills.
3. The task is routed through a direct workflow, script, skill, or subagent pass.
4. Risky actions are checked against policy before execution.
5. The result is returned as one complete message.
6. Useful decisions are written back into memory, docs, or a workflow tracker.

That loop turns the assistant from a disposable conversation into a small operations layer for repeated work.

## Proof modules

### Runtime guardrails

Runtime changes follow a strict policy:

- Only specific runtime files may be edited
- Restart requests go through one approved request path
- Destructive command classes are hard-denied
- Secrets and auth files are never edited
- Failed restart results stop the workflow instead of triggering automatic retries

This is less flashy than "an AI agent fixes itself", but it is much closer to how a trustworthy system should behave.

### Memory as product infrastructure

Memory is split by purpose.

Stable preferences live in durable memory. Daily work goes into date-based logs. Project notes capture current status, constraints, decisions, open risks, and next actions.

That separation matters because a long-term preference and a temporary troubleshooting note should not carry the same weight. The structure lets the assistant avoid re-learning the same context without turning memory into a messy pile of old chat.

### Skills instead of prompt soup

When a workflow repeats, it becomes a skill.

Media import triage, subtitle diagnosis, OpenClaw update handling, reminder tone, button-based approvals, and case-study writing all have defined procedures. A skill describes the trigger, inputs, limits, validator checks, and expected output.

That makes the behavior inspectable. The assistant is not improvising from a giant prompt. It is following a small product system.

### Human-in-the-loop review

One recurring workflow prepares old club archive material for Instagram.

The source material is messy: old Word documents, inconsistent filenames, embedded images, incomplete text extraction, and candidates that need a human decision before public use.

The assistant turns that into a review queue:

- Find candidates that have not been approved, rejected, or posted
- Extract useful text and participant names where possible
- Locate embedded or related images
- Generate a Telegram preview
- Handle approve, reject, and postpone actions through buttons
- Write the approval state before showing the next candidate

The UX decision is deliberately narrow: one candidate at a time, one clear action, state written before the next prompt.

### Subagent-assisted writing and review

This case study was also created through the system it describes.

Instead of producing one generic draft, the workflow used separate passes: gather evidence, identify proof modules, check privacy risk, shape the recruiter narrative, build the page, verify the static export, and deploy through the actual GitHub Pages path.

The first release exposed a process miss: the branch had been pushed, but the live site deployed from `main`. That became a release rule:

- Check the deploy workflow before claiming something is live
- Keep unrelated repo changes in separate commits
- Build against the branch that will actually deploy
- Verify the public URL after local export and CI deployment

That is the kind of feedback loop I want from AI operations: faster detection, cleaner recovery, and better rules after each miss.

## Design decisions

### Make autonomy bounded

Low-risk work should move forward without constant supervision. High-impact actions should not.

The assistant can inspect files, draft text, run checks, and prepare previews. It cannot casually edit secrets, retry failed runtime changes, delete files from triage alone, or post publicly without the right approval path.

### Use ordinary interfaces

The interface is intentionally simple. Telegram is enough for requests, approvals, status checks, and finished outputs.

That matters because the assistant fits into an existing workflow instead of asking for a new control surface every time a task appears.

### Keep the system inspectable

Useful automation needs receipts.

The project uses logs, state files, memory notes, docs, and validation steps so the assistant can explain what happened, what changed, and what still needs attention.

### Treat personality as interaction design

The assistant has operating rules for tone and behavior: be concise, challenge weak assumptions, avoid exposing private details, write follow-ups into durable trackers, and check local evidence before claiming something is unavailable.

That is not cosmetic. A reliable assistant needs a stable interaction style as much as it needs technical capability.

## Impact

The main impact is continuity.

OpenClaw reduces the cost of returning to a task after days or weeks because the context has somewhere to live. It turns repeated work into named workflows. It catches weak assumptions by checking local evidence. It keeps high-risk actions behind guardrails. And it lets me move between design, writing, automation, and operations without starting from zero every time.

For a portfolio, the project shows a grounded AI capability:

- Interfaces matter
- Memory needs structure
- Automation needs constraints
- Human approval is a feature, not a failure
- The best workflows are repeatable, inspectable, and boring in the right way

The result is not just a personal assistant. It is a practical example of designing the conditions where AI can be useful without becoming sloppy.

## What's next

The next step is to make active workflows more visible without making the system noisier.

I want to add a clearer dashboard for current tasks, better visual summaries for scheduled checks, and more reusable subagent roles for research, review, and implementation work.
