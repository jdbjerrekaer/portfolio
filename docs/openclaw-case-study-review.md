# OpenClaw case study review checklist

Use this for live review before treating the case study as finished.

## Recruiter scan

- Does the opening explain why this is more than "I use AI"?
- Is Jonatan's role clear within the first screen?
- Does the case study show product thinking, UX engineering, systems thinking, and delivery?
- Are proof modules concrete enough without exposing private operational details?
- Is the impact credible without inventing metrics?

## Content quality

- Cut any sentence that sounds like generic AI/product filler.
- Keep the narrative focused on designed systems, not tool enthusiasm.
- Prefer concrete workflow decisions over broad claims.
- Keep subagents framed as an orchestration pattern, not a magic trick.
- Avoid mentioning resume, CV, or job-application assistance.

## Live site check

- Verify `/projects/openclaw-ai-assistant/` returns HTTP 200.
- Check desktop and mobile layout.
- Confirm the case study appears on `/projects/`.
- Confirm the cover image loads with the GitHub Pages base path.
- Confirm the GitHub Pages workflow passed after pushing `main`.

## Next iteration candidates

- Replace the generated cover with a more polished visual system if it feels too synthetic.
- Add one compact workflow diagram for request -> evidence -> review -> deploy.
- Add a clearer "before and after" example for the Instagram archive workflow.
- Tighten the ARR/media section if it reads too internal.
- Add stronger numbers only when they can be sourced safely.
